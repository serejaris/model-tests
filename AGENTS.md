# AGENTS.md — правила репозитория model-tests

Этот репозиторий каталогизирует **тестовые треки по моделям**.
Каждый трек — одна модель под тестом. Внутри трека каждый тест
самодостаточен: промпт, который его породил, исходный код,
**PRD**, скриншоты и документация лежат вместе.

## Структура верхнего уровня

```
model-tests/
├── AGENTS.md                 # этот файл
├── README.md                 # обзор репозитория
└── <имя-модели>/             # одна папка на каждую модель
    ├── README.md             # что за модель, что тестируем
    ├── prompts/              # сырые user-промпты к этой модели
    │   └── <NN-slug>/
    │       └── prompt.md     # дословный промпт + короткий контекст
    └── <имя-теста>/          # одна папка на каждый отдельный тест
        ├── README.md         # документация конкретного теста
        ├── PRD.md            # ОБЯЗАТЕЛЬНО для нетривиальных тестов
        ├── screenshots/      # ОБЯЗАТЕЛЬНО для фронтенд-приложений
        │   ├── 01-*.png
        │   └── …
        ├── src/              # ИЛИ исходники прямо на этом уровне
        └── …
```

`<имя-модели>` — каноническое имя модели (например `minimax-m3`,
`claude-opus-4`, `gpt-5`, `gemini-3-pro`). Нижний регистр,
kebab-case, без указания версии в имени папки — версия пишется в
README.

## Правила по треку

1. **Одна папка на модель** — никогда не вкладывать модель в модель.
2. **`prompts/`** — одна подпапка на каждый промпт, который породил
   артефакт. Файл называется `prompt.md` и содержит **дословный**
   user-промпт (без редактуры). Префикс — нумерованная последовательность
   с ведущим нулём (`01-…`, `02-…`), чтобы сохранять хронологический
   порядок при сортировке в файловой системе.
3. **`<имя-теста>/`** — одна папка на конкретный артефакт, который
   модель выдала по промпту. Папка должна быть запускаемой и
   обозримой сама по себе:
   - Свой `README.md` с: что это, как запустить, что тестировали,
     результаты smoke-проверки, известные проблемы.
   - **Свой `PRD.md`** — Product Requirements Document, описывающий
     цели, scope, метрики успеха, ограничения и non-goals. Формат —
     подразделы ниже в разделе «Формат PRD».
   - Свои исходники. Если тест — web-app или игра, то предпочтительно
     `src/` плюс корневые конфиги (`package.json`, `index.html`,
     build-конфиг). Если артефакт — один файл, файл лежит в корне
     теста.
   - Если в тесте несколько под-артефактов (например `frontend/` и
     `backend/`), каждый живёт в своей подпапке внутри папки теста.
4. **`README.md` на каждом уровне** — корень репо, папка модели,
   каждая папка теста. Никаких недокументированных папок.
5. **`screenshots/` + секция «Скриншоты» в `README.md` — обязательны
   для фронтенд-приложений.** Любая UI-штука (web, игра,
   дизайн-тул, бот-интерфейс, мобильный layout и т.п.) **обязана**
   содержать и папку `screenshots/` с PNG-файлами, **и** секцию
   `## Скриншоты` в корневом `README.md` теста с этими картинками,
   встроенными через `![…](screenshots/…)`. Без этой секции тест
   считается незавершённым. Подробно в разделе «Скриншоты».
6. **PR-цикл через issue** — каждый тест стартует с issue и
   заканчивается обновлением этого issue. Подробно в разделе
   «PR-цикл через issue».

## Изоляция работы — своя папка, свой PRD, свой план

Это **главное правило** этой секции. Перед тем, как начинать новый
тест, агент ОБЯЗАН:

1. **Создать собственную папку** под тест
   (`<модель>/<имя-теста>/`) и работать **только внутри неё**. Не
   трогать чужие папки других моделей и других тестов той же модели.

2. **Не читать чужие файлы** (исходники, PRD'ы, README, src/) до тех
   пор, пока не написан собственный PRD. Чтение чужих решений до
   планирования — это **contaminating priors**: PRD и архитектура
   начнут «подстраиваться» под увиденное, вместо того чтобы
   отвечать на свой промпт. Исключение — публичные
   high-level-обзоры (корневой `README.md`, README родительской
   модели, AGENTS.md репо) и **только** для понимания конвенций
   репо. Сами артефакты (`src/`, PRD, скриншоты других тестов) —
   не трогать до завершения собственного PRD.

3. **Написать PRD** в собственной папке **до** любого кода. PRD — это
   стоп-кран: пока не описано «что строим и как поймём, что
   получилось», нет смысла трогать код. Формат — в разделе
   «Формат PRD» ниже.

4. **Запланировать работу** — после PRD, до кода, составить явный
   план. Минимум:

   - Список задач в порядке выполнения (что сначала, что потом).
   - Где граница MVP и что в MVP не входит.
   - Что считаем «готово» — ссылка на метрики успеха из PRD.
   - Что может пойти не так, и какой план Б.

   План кладётся либо отдельным файлом `PLAN.md` в корне теста, либо
   секцией «План работы» в `PRD.md` (формат ниже).

5. **Только после плана** начинать писать код. Каждый коммит несёт
   `Refs: serejaris/model-tests#<issue>` на свой issue.

Если ты пришёл в чужой тест (например, чтобы что-то починить), это
другая ситуация — явно пометь в issue «hotfix: <что сломалось>» и
не считай это новым тестом.

## PRD — обязательно перед стартом работы

Прежде чем писать код по новому тесту, **сначала собирается PRD** и
кладётся в `<модель>/<тест>/PRD.md`. Это не бюрократия — это
форс-мажорный стоп-кран: пока не описано, что именно мы строим и
как поймём, что получилось, нет смысла трогать код.

### Когда PRD обязателен

- Любой нетривиальный тест (больше одного файла исходников или
  дольше 30 минут работы).
- Любой пользовательский интерфейс.
- Любой артефакт, который будет показан людям.

Если тест крошечный (одностраничный HTML без логики, 5 строк
Python), можно обойтись короткой записью в `README.md` и явно
пометить «PRD: n/a — single-screen toy».

### Формат PRD

Файл `PRD.md` в корне теста. Заголовок `# PRD — <имя-теста>`.
Обязательные секции:

1. **Контекст и цель** — какую задачу решаем, для кого, почему
   сейчас.
2. **Scope (in / out)** — что входит, что не входит.
3. **Пользовательские сценарии** — 2–5 коротких сценариев
   «юзер делает X → получает Y».
4. **Метрики успеха** — что измеряем, чтобы понять, что тест
   прошёл. Например: «FPS ≥ 30 на mid-tier ноуте», «smoke.mjs 5/5
   PASS», «cold-start до первого кадра < 2с».
5. **Технические ограничения** — стек, целевые платформы, бюджет
   памяти, и т.д.
6. **Non-goals** — что мы сознательно не делаем, чтобы scope не
   разъехался.
7. **План работы** — обязателен. Внутри теста — это не «документ
   ради документа», а способ зафиксировать, в каком порядке мы
   идём. Минимум: список задач по порядку, граница MVP, критерий
   «готово» (ссылка на метрики успеха), известные риски и план Б.
   Можно жить в `PRD.md` секцией или отдельным `PLAN.md`.
8. **Открытые вопросы** — что нужно решить до / во время работы.

PRD обновляется по мере появления ответов. Финальная версия —
после завершения теста, чтобы зафиксировать, что в реальности
получилось.

## Скриншоты — обязательно для фронтенд-приложений

Любой UI-артефакт (web-приложение, игра, визуальный бот, бот с
miniapp-интерфейсом, мобильный layout и т.п.) **обязан** иметь:

1. Папку `screenshots/` с PNG-файлами как минимум для:
   - **Стартовый экран** — что юзер видит первым.
   - **Основной сценарий** — кадр в середине типичного использования
     (в игре — игровой момент; в приложении — рабочий стол с
     данными).
   - **Дополнительные состояния** — модалки, ошибки, win/lose в
     игре, пустые состояния.

2. Секцию `## Скриншоты` в **корневом `README.md` теста** со
   встроенными картинками через `![…](screenshots/…)`. Секция
   ставится **в верхней трети** README — сразу после метаданных
   и ссылки на PRD, до секции «как запустить». Это лицо теста,
   должно сразу бросаться в глаза.

### Правила съёмки

- **Формат**: PNG, имя `<NN>-<краткое-имя>.png` (`01-start.png`,
  `02-gameplay.png`, `03-victory.png`).
- **Разрешение**: реальное окно приложения (не дамп одной сцены
  Three.js), `≥ 1280×720`.
- **Кто снимает**: headless Puppeteer / Chrome DevTools Protocol —
  не вручную. Если руками — это исключение, и в README нужно
  описать, как воспроизвести кадр.
- **Композиция кадра**: естественная — что увидит реальный
  юзер. Не надо ставить камеру в идеальную точку, если в игре
  такой точки не существует.
- **Когда обновлять**: при любом визуально значимом изменении.
  Старые скрины удалять не нужно, если они всё ещё
  репрезентативны; иначе переснимать с тем же именем.

### Где ещё видны скрины

- В соответствующем GitHub issue — ссылки через raw.githubusercontent,
  чтобы картинки рендерились прямо в тикете.

## PR-цикл через issue

Каждый тест живёт в своём GitHub issue (см. «Кросс-репо трекинг»
ниже). Работа идёт по такому циклу:

1. **Открыть issue с PRD-блоком** — до начала кода. Тело issue
   содержит PRD целиком (или ссылку на `PRD.md` в репо + краткое
   резюме). Issue ставится в `In progress`.
2. **Писать код** — коммиты несут `Refs: <repo>#<issue>`.
3. **Обновить issue** — на каждой значимой вехе (новая механика,
   smoke зелёный, скрины добавлены) обновлять body / добавлять
   комментарии со ссылками на коммиты и скрины. Это work-record
   по iron invariant 6 skill `manager`.
4. **Закрыть issue** — когда smoke зелёный, скрины на месте, PRD
   отражает финальный scope.

## Кросс-репо трекинг

Каждый тест в этом репо ОБЯЗАН также быть задокументирован как
GitHub issue в проекте `ris © corp` (#4) по правилам skill `manager`:

- Один **epic** issue на каждый трек модели, именованный
  `<модель> — testing track` (например `minimax-m3 — testing track`).
- Один **child** issue на каждую папку теста, именованный
  `<модель> — <тест> (<короткий контекст>)`.
- Тело child-issue содержит:
  - PRD-блок (полный текст или ссылка + краткое резюме);
  - ссылки на скриншоты через raw.githubusercontent;
  - ссылку на папку теста (`<модель>/<тест>/`);
  - ссылку на промпт, который его породил;
  - доказательства smoke-проверки.
- W-label + Sub-issues parent link + Project placement по железным
  инвариантам `~/.claude/skills/manager`.

## Как добавить новый тест (workflow)

1. Выбрать папку модели. Если модели ещё нет в репо — создать с
   README.
2. Создать `prompts/<NN-slug>/prompt.md` и вставить дословный
   user-промпт.
3. **Не подглядывая в чужие тесты**, создать свою папку
   `<имя-теста>/` и написать в ней `PRD.md` по формату выше. Это
   блокирующий шаг — код не пишем до готового PRD.
4. Запланировать работу: либо секция «План работы» в `PRD.md`,
   либо отдельный `PLAN.md` в корне теста. См. «Изоляция работы».
5. Открыть GitHub issue с PRD в body, поставить в `In progress`.
6. Положить в `<имя-теста>/` исходники. Добавить `README.md`.
7. Если это фронтенд — снять скриншоты в `screenshots/`.
8. **Встроить скриншоты в `README.md` теста** — секция `## Скриншоты`
   в верхней трети, с `![…](screenshots/…)` для каждого PNG.
9. Прогнать smoke-проверку. Зафиксировать результаты в README
   теста.
10. `git add` + коммит `feat(model-tests): add <модель>/<тест>` и
    push. В коммит-сообщении указать `Refs: serejaris/model-tests#<N>`.
11. Обновить issue: ссылка на коммит, скрины, финальный smoke.
12. Закрыть issue (или оставить открытым, если итерации
    продолжаются — тогда новый issue на следующий шаг).

## Что НЕ должно сюда попадать

- Приватные логи разговоров, телеметрия, всё что под NDA.
- Сгенерированный шум (lock-файлы, `node_modules`, `dist/`,
  `__pycache__`). В каждой папке теста должен быть свой `.gitignore`,
  который это отбрасывает.
- Дубли одного и того же артефакта в нескольких папках тестов.
  Должна быть одна каноническая локация, а в соседних папках — ссылка
  на неё, если нужно.
- Скриншоты, снятые с нестандартного разрешения, без HUD-элементов,
  из «идеальных» ракурсов, которые нельзя воспроизвести из игры.
