# Тест 03 — multiplayer PvE FPS (autonomous full-stack)

Тяжёлый агентный тест: довести **сетевую игру end-to-end до живого
URL**. На входе — подробный промпт автономному full-stack агенту с
root-доступом к чистому Ubuntu VPS. На выходе — playable vertical
slice браузерной co-op PvE survival FPS: authoritative-сервер на
Rust, realtime по WebSocket, браузерный 3D-клиент, волны с боссом,
тесты и развёрнутый сервис с публичной ссылкой.

Это не «нарисуй 3D-сцену» (см. тест 02), а «подними и задеплой
сетевую систему, в которой сервер — источник истины боя».

## Вход

Дословный промпт в [`prompt.md`](./prompt.md). Контекст и заметки —
в том же файле.

## Чем отличается от теста 02

| | 02 — 3d-shooter-game | 03 — multiplayer PvE FPS |
|---|---|---|
| Промпт | 3 слова, расплывчатый | подробный, max-spec |
| Что меряем | додумывание при минимуме ТЗ | удержание большого scope + деплой |
| Игроки | сингл-плеер | 2–8 в одной комнате |
| Бэкенд | не нужен | **Rust, authoritative** |
| Сеть | нет | **WebSocket realtime** |
| Деплой | локальный запуск | **публичный URL на VPS** |
| Стек | полностью свободный | частично фиксирован |
| Среда прогона | можно локально | **только контейнер/VM** |

## Что задано жёстко / что свободно

**Задано (нарушение = провал):**
- сервер на **Rust** и **authoritative** для боя;
- **realtime из браузера** (WebSocket или эквивалент);
- браузерный **3D**-клиент по **одному URL**;
- **мультиплеер** 2–8 игроков в общей комнате, видят друг друга;
- автономный **деплой** до публичной ссылки, переживающей рестарт.

**Свободно (выбор модели):**
- 3D-библиотека (Three.js / Babylon.js / WebGPU / иное);
- оружие (rifle / SMG / shotgun) и набор апгрейдов;
- визуальный стиль (low-poly / neon / sci-fi);
- протокол (JSON или binary);
- одна комната против пути к нескольким комнатам;
- enemy AI и процедурная арена.

## Ожидаемый артефакт

Рабочая клиент-серверная система (см. `prompt.md` → «Артефакт»):
- Rust cargo-сервер: authoritative game loop + WebSocket + раздача
  клиента (или совместный деплой с reverse proxy);
- браузерный 3D-клиент: сцена, FPS-управление, другие игроки и
  враги, стрельба, HUD, экраны входа/смерти;
- Rust-тесты (wave scaling, damage/HP/death/score, wave→boss→next,
  spawn/respawn) + HTTP/WS smoke;
- развёрнутый сервис на VPS с публичным URL и автозапуском.

## Критерии оценки

В [`criteria.md`](./criteria.md). Единый чеклист для всех моделей,
заякоренный на 14 пунктов «Минимального критерия готовности» из
промпта. Группы: структурные, сетевые/authoritative (ядро), геймплей,
деплой, тесты, антипаттерны, бонусы, smoke.

## Как запустить тест на новой модели

> **Важно**: готовность требует **публичный URL** и выживание после
> рестарта сервиса. Значит тест **обязан** идти в реальной среде —
> чистый Ubuntu VPS с root или approval-gated CT/VM (на нашей
> инфраструктуре — по правилам runbook'ов `corp-server`, не локально).

1. Подготовь целевую среду: чистый Ubuntu VPS / CT с root-доступом,
   открытый порт, при наличии — домен.
2. Запусти тестируемую модель/агента **внутри этой среды**.
3. Отправь дословный текст из `prompt.md` **без переписывания**.
4. Проследи порядок: сначала `PRD.md`, затем `PLAN.md`, `TASKS.md`,
   только потом код (правило изоляции из AGENTS.md). Оператор PRD не
   пишет — его пишет модель. Нарушение порядка → `invalid-model-prd`.
5. Дай модели реализовать сервер + клиент + тесты и задеплоить.
6. Прогони `criteria.md` по результату; зафиксируй оценку.
7. Положи артефакт в основной трек модели
   (`<model>/<test>/`), не в `examples/` (см. правило ниже).
8. Сними скриншоты (меню, два игрока, стрельба по волне, boss bar,
   смерть) через headless-клиент, положи в `screenshots/`.
9. Оператор пишет `RUN.md` (среда, контейнер, токены, время,
   preview URL, команды, smoke) и `EVALUATION.md` (pass/fail по
   criteria, что модель сделала сама, нарушения правил, статус).
10. Создай child issue от трека модели по правилам `manager`.

> Артефакт реального прогона живёт в основном треке модели
> (`<model>/<test>/…`), а в `examples/` кладутся только результаты
> чистых сравнительных прогонов — см. `examples/README.md`.

## Чего этот тест НЕ измеряет

- Выбор 3D-библиотеки — это часть свободы.
- Качество графики (всё процедурно).
- Netcode-изощрённость (lag comp / rollback не требуются для v1).
- Производительность под >8 игроков и горизонтальное масштабирование.
- Персистентность / БД (in-memory достаточно).
