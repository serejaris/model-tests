# Критерии оценки — multiplayer PvE FPS

Эти критерии используются, чтобы сравнить, как разные модели
выполняют один и тот же тест. Чем больше критериев закрыто — тем
лучше артефакт.

> **Важно**: в отличие от теста 02, промпт здесь **подробный и
> частично фиксирует стек** (Rust authoritative-сервер, realtime из
> браузера, 3D-клиент, мультиплеер, деплой до публичного URL). Эти
> границы — **must-have**, их нарушение = провал, а не «выбор
> модели». Свобода остаётся в библиотеке клиента, оружии,
> апгрейдах, визуальном стиле и протоколе (см. `prompt.md` →
> «Что промпт оставляет на усмотрение модели»).

Якорь оценки — 14 пунктов «Минимального критерия готовности» из
промпта. Ниже они разложены по группам плюс антипаттерны, бонусы и
smoke.

## Структурные (must-have)

- [ ] **Rust cargo-проект сервера** с понятной структурой и
      разделением доменов: networking, game state, waves, enemies,
      combat, static serving/config (промпт это требует явно).
- [ ] **`cargo build --release` проходит чисто** — без ошибок и без
      падения на варнингах-как-ошибках.
- [ ] **`cargo test` проходит** — тесты ниже (раздел «Тесты»)
      реально существуют и зелёные.
- [ ] **Один URL открывает игру** — сервер раздаёт клиент сам
      (static serving из Rust) или развёрнут с reverse proxy; не два
      разных порта/хоста для «зайти поиграть».
- [ ] **Клиентский код читаем**: обработчик входящих WS-сообщений,
      FPS-контроллер и рендер-цикл находятся без grep по одному
      мегафайлу (логически выделённые файлы/модули).
- [ ] **Без внешних платных сервисов, секретов, API-ключей,
      кошельков** — in-memory state достаточно для v1.
- [ ] **Без приватных данных** в репо (`.env` с ключами,
      credentials, токены).

## Сетевые / authoritative (must-have — ядро теста)

- [ ] **Сервер authoritative для боя** (готовность №12): HP, урон,
      убийства, очки, волна, спавн врагов, апгрейды и смерть живут
      на сервере. Клиент шлёт **намерения** (move/look/shoot), не
      результаты.
- [ ] **Клиенту не доверяется** урон/HP/убийства/очки/волна. В коде
      нет пути, где клиентское сообщение напрямую задаёт чужой HP
      или начисляет себе очки.
- [ ] **2–8 игроков в одной комнате** (готовность №3): несколько
      WebSocket-подключений попадают в общий мир.
- [ ] **Игроки видят друг друга** (готовность №4) и **движение
      синхронизируется** между клиентами.
- [ ] **Выстрелы, попадания, убийства и спавн врагов видны всем**
      игрокам, а не только стрелявшему.
- [ ] **Realtime-цикл сервера** (tick loop, 20–30 Hz для v1)
      рассылает состояние мира подключённым клиентам.

## Геймплей (must-have)

- [ ] **FPS-управление + стрельба** (готовность №5): WASD + мышь
      (pointer-lock), выстрел.
- [ ] **Минимум один огнестрел** (rifle / SMG / shotgun) с
      серверным расчётом попадания.
- [ ] **Враги волнами** (готовность №6): спавн по волнам, не
      статичная россыпь.
- [ ] **Враги идут на игроков и атакуют** (готовность №7) — вблизи
      или на дистанции, не «статуи».
- [ ] **У врагов есть HP, они умирают от выстрелов** (готовность
      №8) с понятным feedback.
- [ ] **Сложность растёт** со временем и/или числом игроков
      (готовность №9) — больше игроков → плотнее/агрессивнее волны.
- [ ] **Хотя бы один boss wave** (готовность №10): босс отличается
      от обычных врагов (HP, размер, атака, фаза или спецэффект).
- [ ] **HUD** (готовность №11): HP, wave, score/kills, players.
- [ ] **Смерть и respawn** игрока обрабатываются сервером.

## Деплой / автономность (must-have)

- [ ] **Production build собран и сервис запущен** на VPS.
- [ ] **Публичная ссылка** открывается из браузера (готовность №1,
      №2) — домен или `IP:port`.
- [ ] **Автозапуск через systemd** (готовность №13): реальный unit
      (корректные `ExecStart`, `WorkingDirectory`,
      `Restart=on-failure`), а не `tmux`/`screen`/`nohup` под видом
      сервиса.
- [ ] **Выживание после рестарта**: `systemctl restart` → (a)
      `systemctl is-active` = `active`; (b) `GET /` отдаёт 200 за
      разумное время; (c) новый WebSocket-коннект снова получает
      `init`/`state`. Сброс in-memory прогресса при этом ожидаем и
      провалом не считается.
- [ ] **Никаких ручных шагов от пользователя**, кроме открытия
      ссылки.
- [ ] **Финальный отчёт содержит URL** (готовность №14), путь к
      проекту, команду рестарта, список тестов и их результат.

## Тесты (must-have — промпт требует явно)

Минимальный набор Rust-тестов должен существовать и быть зелёным:

- [ ] **scaling волн** по числу игроков (unit) — *ядро, обязателен от 4/5*.
- [ ] **урон / HP / death / score** (unit) — *ядро, обязателен от 4/5*.
- [ ] **переход wave → boss → next wave** (unit/логика стейта) — *ядро, обязателен от 4/5*.
- [ ] **spawn / respawn** базовой логики (unit).
- [ ] **HTTP smoke**: клиентская страница отдаётся (200 + контент).
- [ ] **WebSocket smoke**: клиент подключается, получает init/state.
- [ ] *(бонус)* integration с двумя mock-клиентами в одной комнате.

## Антипаттерны (минусы / провал, если найдены)

- [ ] **Заглушка вместо игры** — промпт запрещает явно.
- [ ] **Только backend без клиента** или **только клиент без
      сервера** — провал.
- [ ] **Красивая сцена без мультиплеера** (сингл-плеер демо под
      видом сетевой игры) — провал.
- [ ] **Клиент-authoritative бой** — урон/HP/убийства считаются на
      клиенте и сервер им верит — провал по ядру теста.
- [ ] **Сервер не на Rust** — провал по готовности №12.
- [ ] **Требуются ручные действия пользователя** после запуска,
      кроме открытия ссылки.
- [ ] **`tmux` / `screen` / `nohup` под видом systemd** — бинарь
      крутится в сессии, а выдан за сервис → провал деплойного
      требования (готовность №13).
- [ ] **Платные сервисы / внешние ключи / кошельки / крипта** в
      контуре игры.
- [ ] **Загрузка внешних 3D-моделей / текстур** (`loadAsync('*.glb')`)
      — графика должна быть процедурной.
- [ ] **TODO-заглушки** в ключевых местах (пустой combat tick,
      пустой spawn). TODO допустим только в плане, не в горячем пути.

## Визуальные / качественные (бонус)

- [ ] **Осознанный стиль** (low-poly / neon / sci-fi / arena) с
      согласованным освещением и палитрой.
- [ ] **Чёткие enemy types**: swarm / tank / ranged / boss с
      различимыми силуэтами.
- [ ] **Импакт-эффекты**: muzzle flash, tracers, искры/декали при
      попадании, death effects.
- [ ] **Health bars / damage indicators** над врагами или игроками.
- [ ] **Boss health bar** отдельным элементом HUD.
- [ ] **Экран смерти + respawn flow** и **меню входа** (ник или
      случайный).
- [ ] **Таблица игроков** (scoreboard).
- [ ] **Апгрейды оружия** (2–4): damage, fire rate, reload, spread,
      piercing, explosive, lifesteal, move speed.
- [ ] **Client-side interpolation** других игроков и врагов (плавно,
      без телепортов между тиками).
- [ ] **Звук через WebAudio** (синтез, без аудио-файлов).
- [ ] **Путь к нескольким комнатам** заложен в архитектуре (даже
      если v1 — одна комната).

## Скриншоты (обязательны для фронтенда)

- Меню входа (ник / случайный).
- Арена с двумя игроками, видящими друг друга.
- Момент стрельбы по волне врагов (muzzle flash / tracer).
- Boss wave с boss health bar.
- Экран смерти / respawn.

## Как оценивать

- **5/5** — все структурные + все сетевые/authoritative + все
  геймплейные + деплой с переживанием рестарта + все 6 обязательных
  тестов зелёные (integration с 2 mock-клиентами не обязателен, но
  засчитывается в бонусы) + 0 антипаттернов + ≥4 бонусных.
- **4/5** — все структурные + все сетевые/authoritative + ≥7
  геймплейных + деплой с публичным URL + ≥5 тестов из набора, **в
  т.ч. обязательно три ядровых** (scaling волн, wave→boss→next,
  урон/HP/death/score) + 0 критичных антипаттернов + ≥2 бонусных.
- **3/5** — сервер authoritative на Rust + мультиплеер из 2 вкладок
  (оба игрока видят друг друга, готовность №3+№4) + волны + хотя бы
  1 boss + HUD + публичный URL, но часть
  тестов/деплой-устойчивости/polish недотянута.
- **≤2** — любой из «провальных» антипаттернов (нет мультиплеера,
  клиент-authoritative бой, нет сервера на Rust, нет публичного URL,
  заглушка) → тест не пройден, нужен ретрай.

## Smoke-проверка (минимум)

Оператор подтверждает на развёрнутом сервисе:

1. **HTTP**: `GET /` отдаёт клиентскую страницу (200 + HTML с
   точкой входа клиента).
2. **WebSocket**: клиент подключается и получает `init`/`state`
   сообщение от сервера.
3. **Authoritative бой**: имитация выстрела через сервер уменьшает
   HP врага; враг умирает → счёт растёт на сервере.
4. **Authoritative не доверяет клиенту** (ключевой механический
   чек): отправить серверу поддельные WS-сообщения напрямую —
   `{type:"set_hp", enemy_id, hp:0}` и `{type:"add_score",
   amount:9999}`. Сервер обязан их игнорировать: враг не умирает,
   счёт не меняется. Если меняется — провал по ядру теста.
5. **Wave scaling** (Rust unit): при росте числа игроков параметры
   волны (плотность/HP/спавн-рейт) увеличиваются.
6. **Wave → boss → next** (Rust unit): достигается boss-волна, после
   её зачистки стейт переходит к следующей волне.
7. **Две вкладки** (предпочтительно автоматизируемо через
   2-mock-client integration; иначе — ручная проверка оператором,
   тогда помечается как non-CI шаг): два клиента в одной комнате
   видят движение и выстрелы друг друга.
8. **Рестарт сервиса**: `systemctl restart <unit>` → `systemctl
   is-active` = `active`; `GET /` снова 200; новый WS-коннект снова
   получает `init`/`state`.
9. **Rust-тесты**: `cargo test` → 0 fail.
10. **Headless-клиент** (если есть инструмент: `chromium --headless`
    / Playwright / Puppeteer открывает URL): страница рендерится, 0
    критичных `console.error` (Three.js `DEPRECATED`-варнинги не
    считаются).

## Чего этот тест НЕ измеряет

- Какую именно 3D-библиотеку выбрала модель (Three / Babylon /
  WebGPU) — это часть свободы.
- Качество графики / текстур (всё процедурно, оценка — согласованность,
  не фотореализм).
- Сетевую изощрённость: lag compensation, rollback, client-side
  prediction уровня соревновательного шутера **не требуются** для v1.
- Производительность под >8 игроков и горизонтальное масштабирование.
- Персистентность / БД (для v1 in-memory достаточно).
- Балансировку геймплея и «весёлость» — достаточно, чтобы петля
  spawn→kill→wave→boss работала.
