# Промпт 03 — multiplayer PvE FPS (autonomous full-stack)

Дословный user-промпт, описывающий общий тест «браузерная co-op PvE
survival FPS с authoritative-сервером». Запускается на любой модели,
способной вести full-stack проект автономно: спроектировать,
реализовать, протестировать, **задеплоить на чистый VPS** и выдать
рабочую публичную ссылку.

В отличие от `02-3d-shooter-game` (трёхсловный расплывчатый промпт,
сингл-плеер, стек свободный) — здесь промпт **максимально подробный**
и фиксирует ключевые границы: authoritative-бэкенд на Rust, realtime
по WebSocket, браузерный 3D-клиент, мультиплеер, автономный деплой.
Тест меряет способность довести систему **end-to-end до живого URL**,
а не нарисовать сцену.

## Промпт

```text
Ты — автономный senior game/backend/full-stack агент. 
У тебя есть root-доступ к чистому Ubuntu VPS. Нужно самостоятельно спроектировать, реализовать, протестировать, запустить и выдать рабочую ссылку на браузерную игру.

Цель
Создать playable vertical slice браузерной multiplayer PvE FPS-игры: клон Vampire Survivors по ощущению прогрессирующих волн, но от первого лица и с огнестрельным оружием.

Игроки открывают одну ссылку в браузере, вводят ник или получают случайный, подключаются к общей комнате и сразу играют. Установка клиента не требуется.

Главная фантазия
Несколько игроков оказываются на красивой технологичной 3D-арене. Они от первого лица отбиваются от всё более сложных волн врагов. Чем больше игроков подключено, тем агрессивнее и плотнее становятся волны. Через заданные интервалы появляются элитные враги и боссы. Игроки получают очки/опыт/апгрейды, открывают усиления оружия и пытаются выжить как можно дольше.

Жанр и геймплей
- Co-op PvE survival arena.
- First-person shooter.
- Вдохновение: Vampire Survivors, Risk of Rain, Killing Floor, Doom-like arena combat.
- Основной loop: spawn → волна врагов → убийства → опыт/очки → апгрейды → следующая волна → босс → усиление сложности.
- Минимум один основной огнестрельный weapon: rifle/SMG/shotgun на выбор агента.
- Можно добавить 2–4 апгрейда: damage, fire rate, reload speed, spread, piercing, explosive shots, lifesteal, movement speed.
- Враги должны идти на игроков, атаковать вблизи или на дистанции, умирать с понятным feedback.
- Боссы должны отличаться от обычных врагов: больше HP, размер, атака, фаза или спецэффект.
- Игра должна быть понятной за 10 секунд без инструкции.

Технологическое направление
- Сервер: Rust.
- Сервер должен быть authoritative: клиенты отправляют намерения, сервер владеет состоянием мира, HP, уроном, волнами, врагами, счётом, апгрейдами и смертью.
- Сеть: WebSocket или другой realtime-протокол, который можно открыть из браузера.
- Клиент: браузерный 3D, свободный выбор библиотеки. Подойдут Three.js, Babylon.js, WebGPU/WebGL или другой практичный вариант.
- Сервер должен также раздавать клиентские файлы или быть развёрнут вместе с простым static server / reverse proxy.
- Один URL должен открывать игру.
- База данных не обязательна. Для v1 достаточно in-memory состояния.
- Реальные деньги, криптовалюта, токены и внешние кошельки в этой версии не нужны.

Качество реализации
- Приоритет: playable, красивый, технологичный vertical slice.
- 3D-модели и ассеты можно делать процедурно или использовать простые low-poly формы.
- Визуал должен выглядеть аккуратно: освещение, цветовая схема, частицы, muzzle flash, hit feedback, death effects, enemy silhouettes, boss presence, HUD.
- Не тратить время на сложные внешние ассеты, если можно добиться красивого результата процедурной графикой.
- Код должен быть достаточно чистым, чтобы другой агент мог продолжить работу.
- Избегать чрезмерной архитектуры, но отделить основные домены: networking, game state, waves, enemies, combat, static serving/config.

Сетевые требования
- Несколько игроков должны видеть друг друга.
- Движение игроков синхронизируется.
- Выстрелы, попадания, убийства и спавн врагов видны всем.
- Сложность масштабируется по количеству активных игроков.
- Сервер не должен доверять клиенту урон, HP, убийства, очки или волну.
- Клиент может делать визуальные эффекты мгновенно, но итоговое состояние подтверждает сервер.
- Минимальная цель: 2–8 игроков в одной комнате на одном VPS.
- Желательно заложить путь к нескольким комнатам, но v1 может иметь одну комнату.

Тесты и валидация
Начать с короткого плана тестирования и реализовать тесты там, где это разумно.
Минимальный набор:
- unit-тесты Rust-логики для scaling волн по числу игроков;
- тесты урона/HP/death/score;
- тесты перехода wave → boss → next wave;
- тесты spawn/respawn базовой логики;
- smoke-тест HTTP: клиентская страница отдаётся;
- smoke-тест WebSocket: клиент может подключиться, получить init/state;
- если есть время, простой integration test с двумя mock-клиентами.

Тесты не должны блокировать playable-результат. Если визуальную часть сложно покрыть автотестами, сделать smoke/manual checklist.

Деплой
Ты работаешь на чистом Ubuntu VPS с root-доступом.
Самостоятельно:
- проверь окружение;
- установи нужные зависимости;
- установи Rust toolchain;
- создай проект;
- реализуй сервер и клиент;
- собери production build;
- запусти сервис;
- открой нужный порт;
- при возможности настрой systemd service;
- при наличии домена используй его, иначе выдай ссылку по IP и порту;
- если нужен reverse proxy, настрой nginx/caddy;
- убедись, что ссылка открывается из браузера.

Ограничения
- Не останавливаться на плане.
- Не спрашивать уточнений, если можно принять разумное решение.
- Не делать заглушку вместо игры.
- Не делать только backend без клиента.
- Не делать только красивую сцену без multiplayer gameplay.
- Не завязываться на внешние платные сервисы.
- Не использовать секреты, API-ключи или внешние аккаунты.
- Не требовать от пользователя ручных действий после запуска, кроме открытия ссылки.

Минимальный критерий готовности
Игра считается готовой, когда:
1. Есть публичная ссылка.
2. Ссылка открывает браузерный 3D-клиент.
3. Два игрока из разных вкладок могут подключиться.
4. Игроки видят друг друга.
5. Можно двигаться и стрелять.
6. Враги появляются волнами.
7. Враги атакуют игроков.
8. У врагов есть HP, они умирают от выстрелов.
9. Сложность растёт со временем и/или числом игроков.
10. Появляется хотя бы один boss wave.
11. Есть HUD: HP, wave, score/kills, players.
12. Сервер написан на Rust и является источником истины для боя.
13. После перезапуска сервиса игру можно открыть снова.
14. В финальном ответе есть URL.

Желательный уровень polish
- Красивый low-poly / neon / sci-fi / arena visual style.
- Чёткие enemy types: swarm, tank, ranged, boss.
- Понятные эффекты попаданий.
- Muzzle flash и tracers.
- Health bars или damage indicators.
- Экран смерти и respawn.
- Простое меню входа.
- Таблица игроков.
- Boss health bar.
- Звуки можно сгенерировать через WebAudio, без внешних файлов.

Технические подсказки, но не жёсткие требования
- Rust backend: axum / actix-web / warp + tokio.
- WebSocket: axum ws / tokio-tungstenite.
- Static serving из Rust допустим.
- Client: TypeScript/JavaScript + Three.js/Babylon.js.
- Tickrate: 20–30 Hz для v1.
- Client-side interpolation для других игроков и врагов.
- Простые collision/raycast достаточно для vertical slice.
- Enemy AI может быть steering-to-nearest-player.
- Карта может быть процедурной ареной с препятствиями.
- Использовать compact JSON или binary protocol на выбор агента.
- Начать с JSON проще для дебага, потом можно оптимизировать.

Порядок работы
1. Быстро определить архитектуру и директории проекта.
2. Создать Rust server skeleton.
3. Написать тесты для game-state/waves/combat.
4. Реализовать authoritative game loop.
5. Реализовать WebSocket-протокол.
6. Реализовать браузерный 3D-клиент.
7. Добавить enemy waves, scaling, boss.
8. Добавить HUD и визуальный polish.
9. Прогнать тесты.
10. Собрать и запустить production.
11. Проверить двумя вкладками.
12. Настроить автозапуск сервиса.
13. Вернуть финальный отчёт.

Финальный ответ должен содержать
- Рабочий URL.
- Где лежит проект на сервере.
- Как перезапустить сервис.
- Какие тесты были запущены и их результат.
- Что реализовано.
- Что осталось на следующий шаг.
- Короткий список ключевых файлов.
```

## Контекст

- **Зачем**: проверить, как модель ведёт **автономный full-stack
  проект end-to-end** — от архитектуры до живого URL: authoritative
  игровой сервер на Rust, realtime-протокол, браузерный 3D-клиент,
  мультиплеер-синхронизация, волновой геймплей с боссом, тесты и
  деплой на чистый VPS. Это прокси-метрика «может ли агент сам
  поднять сетевую систему на сервере и доказать, что она работает».
- **Кто будет смотреть**: те, кто сравнивает модели на тяжёлых
  агентных задачах (не «нарисуй сцену», а «доведи продукт до
  работающего URL под нагрузкой двух вкладок»).
- **Что считается успехом**: см. `criteria.md`. Якорь — 14 пунктов
  «Минимального критерия готовности» из самого промпта.

## Заметки к дословному промпту

В отличие от теста 02, этот промпт **специально перегружен
деталями**. Идея другая: не «как модель додумывает при минимуме
требований», а «как модель держит большой scope, не срезает углы и
доводит до деплоя». Минусы здесь — это **срезанные углы**: backend
без клиента, клиент без сервера, красивая сцена без мультиплеера,
доверие клиенту в расчёте урона, заглушка вместо игры.

Что промпт **фиксирует жёстко** (нарушение = провал):
- сервер на **Rust** и **authoritative** для боя (готовность №12);
- **realtime из браузера** (WebSocket или эквивалент);
- браузерный **3D**-клиент по **одному URL**;
- **мультиплеер**: 2–8 игроков в одной комнате видят друг друга;
- автономный **деплой** до публичной ссылки, переживающий рестарт
  сервиса (готовность №13).

Что промпт **оставляет на усмотрение модели**:
- 3D-библиотека клиента (Three.js / Babylon.js / WebGPU / иное);
- конкретное оружие (rifle / SMG / shotgun) и набор апгрейдов;
- визуальный стиль (low-poly / neon / sci-fi);
- протокол (JSON или binary);
- одна комната против пути к нескольким комнатам;
- детали enemy AI и процедурной арены.

## Артефакт, который ожидаем

**Не один HTML-файл и не только фронтенд.** Ожидается рабочая
клиент-серверная система:

- Rust-сервер (cargo-проект): authoritative game loop, WebSocket,
  раздача клиентских файлов или совместный деплой со static
  server / reverse proxy.
- Браузерный 3D-клиент: сцена, FPS-управление, рендер других
  игроков и врагов, стрельба, HUD, экраны входа/смерти.
- Тесты Rust-логики (wave scaling, damage/HP/death/score, переход
  wave→boss→next, spawn/respawn) + HTTP/WS smoke.
- Развёрнутый сервис на VPS с публичным URL и автозапуском.

Подробные критерии — в `criteria.md`. Так как готовность требует
**публичный URL и выживание после рестарта**, этот тест по сути
**обязан прогоняться в реальном контейнере/VM**, а не в локальной
сессии модели (важное отличие от теста 02).
